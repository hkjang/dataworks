# Data Works Design Standard

> 상태: Active
>
> 문서 책임: Product Design
>
> 구현 책임: Admin UI Engineering
>
> 적용 범위: Data Works Admin UI 전체

이 문서는 Data Works의 디자인 원칙과 UI 계약을 정의하는 단일 기준 문서다. 구현이 이 문서와 다르면 버그로 취급한다. 의도적으로 다른 디자인을 적용하려면 코드 변경과 같은 변경 단위에서 이 문서도 함께 갱신해야 한다.

## 1. 디자인 목표

Data Works는 데이터, 상품, 플로우, 에이전트, 정책을 반복적으로 운영하는 엔터프라이즈 작업 도구다. 화면은 다음 질문에 빠르게 답해야 한다.

1. 지금 보고 있는 개체는 무엇인가?
2. 현재 상태와 위험은 무엇인가?
3. 어떤 근거와 연결 개체가 사용되는가?
4. 다음에 해야 할 작업은 무엇인가?
5. 실행 또는 변경 결과를 어디에서 확인하는가?

마케팅 페이지처럼 보이는 구성보다 높은 정보 밀도, 예측 가능한 탐색, 비교 가능한 상태, 명확한 작업 흐름을 우선한다.

## 2. 핵심 원칙

### 2.1 개체를 직접 입력하지 않는다

이미 등록된 데이터 자산, 상품, 워크스페이스, 플로우, 에이전트, 도구, 정책, 고객군, 오너는 자유 텍스트로 입력하지 않는다.

- 단일 선택: 검색 가능한 선택 상자
- 복수 선택: 체크박스 기반 선택 목록
- 목록이 긴 경우: 검색, 필터, 선택 모달
- 선택 직후: 이름, 키, 오너, 상태, 위험도, 설명 미리보기
- 비활성 또는 폐기 개체: 기본 선택 목록에서 제외

자유 텍스트는 새 개체의 이름, 설명, 목적, 계약, 프롬프트처럼 새 값을 정의할 때만 사용한다.

### 2.2 상태와 다음 작업을 함께 보여준다

상태 배지만 보여주고 사용자가 의미를 추론하게 하지 않는다. 상세 화면은 상태, 차단 사유, 필요한 승인, 다음 작업을 같은 시야에 둔다.

### 2.3 거버넌스를 작업 흐름 안에 둔다

정책, 승인, 민감도, 준비도, 마스킹, 감사 정보는 별도 보고서에 숨기지 않는다. 선택 미리보기, 실행 전 검증, 출시 게이트에서 즉시 노출한다.

### 2.4 AI 결과는 초안이다

AI 생성 결과는 저장 전에 반드시 미리보기 상태를 거친다.

`생성 요청 -> 진행 상태 -> 미리보기 -> 비교 -> 채택 또는 수정 -> 명시적 저장`

AI 결과를 자동 저장하거나 기존 값을 조용히 덮어쓰지 않는다.

### 2.5 한국어로 판단하고 영어 식별자를 보조로 쓴다

- 메뉴, 버튼, 상태, 검증, 오류, 빈 화면은 한국어로 작성한다.
- API 키, URN, 개체 키, 프로토콜명은 원문을 유지한다.
- 영어 식별자는 이름보다 작은 보조 정보로 표시한다.
- 백엔드의 영문 오류나 정책 사유는 사용자용 한국어로 변환한다.

### 2.6 점진적으로 복잡해진다

첫 화면에는 가장 자주 쓰는 필드와 작업만 보여준다. 스키마, 비용 한도, 원시 계약처럼 전문 설정은 `고급 설정` 아래에 둔다.

## 5. 디자인 책임 모델

| 결정 | 책임자 | 필수 협의 | 승인 기준 |
| --- | --- | --- | --- |
| 정보 구조와 핵심 흐름 | Product Design | Product Owner, Domain Owner | 주요 업무가 중단 없이 완료됨 |
| 용어와 상태 체계 | Product Design | Domain Owner, Backend | API 상태와 사용자 문구가 일관됨 |
| 공통 컴포넌트 | Admin UI Engineering | Product Design | 재사용 가능하고 접근성 계약을 충족함 |
| 정책·위험 표현 | Governance Design Owner | Risk, Compliance | 차단 사유와 개선 조치가 명확함 |
| 데이터 계약과 참조 목록 | Backend Owner | Admin UI Engineering | 활성 개체와 권한이 정확히 반영됨 |
| 반응형·접근성 품질 | Admin UI Engineering | QA | 지원 화면 크기와 키보드 검증 통과 |
| 최종 디자인 승인 | Product Design | Product Owner, QA | 이 문서의 릴리즈 게이트 통과 |

책임자 개인 이름은 조직 변경에 따라 별도 운영 문서에서 관리한다. 이 문서에는 역할을 기록해 책임이 특정 개인에게 고정되지 않게 한다.

## 6. 화면 구조

### 6.1 애플리케이션 셸

- 상단에는 브랜드, 1차 탐색, 사용자 도구를 둔다.
- 현재 영역은 색상과 대비로 명확하게 표시한다.
- 관련 하위 화면은 2차 탭으로 묶는다.
- 본문 최대 폭은 `1440px`이며 큰 화면에서도 정보가 과도하게 퍼지지 않게 한다.
- 운영 도구이므로 대형 히어로, 홍보성 카피, 장식용 배경은 사용하지 않는다.

### 6.2 목록 화면

`제목과 요약 KPI -> 검색·필터 -> 결과 표 -> 빠른 작업`

- 필터 값은 화면 이동 후에도 가능한 범위에서 유지한다.
- 표 행은 이름, 키, 오너, 상태, 수정일을 기본으로 제공한다.
- 행 전체 이동과 행 내부 버튼이 충돌하지 않게 한다.
- 결과가 없으면 원인과 다음 작업을 함께 제시한다.

### 6.3 상세 화면

`개체 헤더 -> 점수와 상태 -> 다음 작업 -> 차단 사유 -> 상세 탭`

- 이름과 상태는 첫 화면에서 가장 강한 식별 신호다.
- 내부 키는 이름 아래 보조 정보로 표시한다.
- 위험 점수는 숫자만 표시하지 않고 의미와 조치를 연결한다.
- 사용자가 저장하지 않은 변경은 명확하게 구분한다.

### 6.4 생성 위저드

- 한 단계에는 하나의 판단만 요구한다.
- 완료 단계, 현재 단계, 이동 가능한 단계를 구분한다.
- 다음 단계로 이동하기 전에 현재 단계의 필수 조건을 검증한다.
- 마지막 단계에서 선택한 개체와 입력 내용을 다시 보여준다.
- 생성 버튼은 검증을 통과하기 전까지 비활성화한다.

### 6.5 스튜디오

데스크톱 기본 구조는 다음과 같다.

`왼쪽 노드 팔레트 | 중앙 캔버스 | 오른쪽 속성 패널`

하단에는 실행 입력, 검증 결과, 실행 로그 또는 트레이스를 둔다.

- 팔레트: 추가 가능한 노드를 업무 범주별로 묶는다.
- 캔버스: 선택 노드와 연결 방향이 명확해야 한다.
- 속성: 등록소 개체는 선택형 입력과 미리보기를 사용한다.
- 실행: 저장 전 로컬 검증과 샘플 실행을 지원한다.
- 모바일: 세 영역을 한 열로 쌓고 표만 내부 가로 스크롤을 허용한다.

## 7. Product Twin 계약

Product Twin은 상품 관련 화면의 기준 상세 화면이다.

### 7.1 헤더 필수 정보

- 상품명과 상품 키
- 현재 상태
- 오너와 제공 유형
- 매출 점수, 리스크 점수, 버전
- 상태에 맞는 다음 작업
- 출시 게이트 결과

### 7.2 탭 순서

`개요 -> 상품 캔버스 -> 근거 -> 리스크 -> API -> 계약 -> PoC -> 제안 -> 비용 -> 전환 퍼널 -> 버전 -> 감사`

탭은 상품 수명주기의 판단 순서를 반영한다. 새 탭은 독립된 업무 판단이 있고 기존 탭으로 설명할 수 없을 때만 추가한다.

### 7.3 출시 게이트

- 성공: 출시 가능 상태와 충족한 근거를 표시한다.
- 실패: 한국어 차단 사유, 누락 승인, 해결 화면 바로가기를 표시한다.
- 출시 버튼: 게이트를 통과하기 전 비활성화한다.
- 차단 사유에 개체 키만 있는 경우에도 개체 유형과 필요한 조치를 함께 설명한다.

## 8. 공통 컴포넌트 계약

### 공통 텍스트 구조

컴포넌트 컨테이너에 사용자 문구를 직접 텍스트 노드로 두지 않는다. 의미 단위별 `span`을 두어 간격, 줄바꿈, 말줄임, 반응형 스타일을 독립적으로 제어한다.

| 문구 유형 | 필수 클래스 |
| --- | --- |
| 섹션 제목 | `.section-title-text` |
| 섹션 설명 | `.section-intro-text` |
| KPI 라벨 / 값 | `.kpi-label-text` / `.kpi-value-text` |
| 키·값 라벨 / 값 | `.kv-label-text` / `.kv-value-text` |
| 개체명 / 개체 설명 | `.entity-title-text` / `.entity-summary-text` |
| 안내 제목 / 본문 | `.message-title-text` / `.message-text` |
| 내부 식별자 / 메타데이터 | `.identifier-text` / `.meta-text` |

표준 키·값 마크업:

```html
<div class="kv">
  <div class="k"><span class="kv-label-text">상품명</span></div>
  <div class="v"><span class="kv-value-text"><strong>SME Watchlist Report</strong></span></div>
  <div class="k"><span class="kv-label-text">상품 키</span></div>
  <div class="v"><span class="kv-value-text"><code>sme_watchlist_report</code></span></div>
</div>
```

표준 섹션 설명 마크업:

```html
<p class="section-intro">
  <span class="section-intro-text">운영자가 즉시 후속 조치해야 할 상품화 경고 및 대기 작업</span>
</p>
```

시각 규격:

- `.section-intro`는 제목 바로 아래의 독립 안내 행으로 배치하고 `10px 14px` 안쪽 여백, 하단 구분선, 보조 배경색을 사용한다. 음수 마진으로 제목에 붙이지 않는다.
- `.kv`는 `라벨 / 값` 2열 정의 목록으로 표시한다. 각 행의 최소 높이는 `40px`이며 라벨과 값 모두 `9px 12px` 안쪽 여백을 갖는다.
- `.kv .k`에는 보조 배경색과 오른쪽 구분선을 적용하고, 각 행 사이에는 하단 구분선을 둔다.
- 좁은 화면에서도 라벨 열은 최소 `104px`을 유지하고 값은 줄바꿈해 문구가 겹치거나 잘리지 않게 한다.
- 상세 카드의 콘텐츠는 `.twin-detail-body`처럼 공통 본문 래퍼 안에 배치하고 데스크톱 `16px`, 모바일 `12px` 안쪽 여백을 적용한다.

- `div`, `p`, `h2`, `strong`, `code`는 구조와 의미를 담당하고 실제 문구는 내부 `span`이 담당한다.
- 버튼, 입력 옵션, 표 셀처럼 네이티브 의미와 상호작용이 우선인 요소는 불필요하게 중첩하지 않는다.
- 단순 값은 반드시 값 전용 span으로 감싼다. 표나 폼 같은 블록 콘텐츠는 값 컨테이너에 직접 배치할 수 있다.
- 동적 렌더링 결과도 동일한 계약을 적용하며 직접 텍스트 노드 검사를 통과해야 한다.

### 8.1 필드

모든 업무 필드는 다음 순서를 따른다.

1. 라벨
2. 짧은 도움말
3. 입력 컨트롤
4. 선택 미리보기 또는 인라인 오류

필수 여부를 placeholder에만 의존하지 않는다. 오류는 문제가 발생한 필드 가까이에 표시한다.

표준 마크업은 다음 구조를 사용한다. 라벨 문자열을 입력 컨트롤에 직접 붙이지 않는다.

```html
<div class="dw-field">
  <label class="field-label required" for="field-id">
    <span class="field-label-text">필드 라벨</span>
  </label>
  <span class="field-help" id="field-id-field-help">입력 기준 설명</span>
  <div class="field-control">
    <input id="field-id" aria-describedby="field-id-field-help field-id-error">
  </div>
  <span class="field-error" id="field-id-error" aria-live="polite"></span>
</div>
```

- 라벨과 도움말은 반드시 별도 `span`으로 분리한다.
- 입력 컨트롤은 `.field-control` 안에 둔다.
- 체크박스와 라디오는 설명을 `.control-copy` span에 둔다.
- 화면별 임의 margin으로 라벨 간격을 만들지 않는다.

### 8.2 참조 선택과 미리보기

미리보기 기본 정보:

- 표시 이름과 내부 키
- 상태와 위험도
- 오너
- 업무 설명
- 자산이면 민감도
- 도구면 승인 필요 여부

선택이 없을 때는 빈 카드 대신 “선택한 항목의 요약이 여기에 표시됩니다”처럼 결과 위치를 예고한다.

### 8.3 상태 배지

| 의미 | 대표 상태 | 표현 |
| --- | --- | --- |
| 정상 | 활성, 완료, 승인됨, 출시됨 | 긍정 배경과 높은 대비 |
| 주의 | 초안, 검토 중, 승인 대기, 보통 위험 | 경고 배경 |
| 차단 | 실패, 차단됨, 폐기됨, 매우 높음 | 오류 배경 |

- 상태는 색상만으로 구분하지 않고 텍스트를 항상 포함한다.
- API 원문 대신 표준 한국어 라벨을 사용한다.
- 같은 상태는 모든 화면에서 같은 문구와 색을 사용한다.

### 8.4 버튼

- 기본 버튼: 화면의 핵심 작업 하나
- 보조 버튼: 비교, 미리보기, 이동, 취소
- 위험 버튼: 삭제, 폐기, 접근 철회
- 아이콘만으로 의미가 명확한 작업은 아이콘 버튼과 tooltip을 사용한다.
- 같은 영역에 기본 버튼을 여러 개 두지 않는다.
- 실행, 출시, 삭제처럼 영향이 큰 작업은 결과와 대상이 명확한 확인 과정을 둔다.

### 8.5 카드와 섹션

- 카드는 반복되는 개별 개체, 모달, 독립 도구에만 사용한다.
- 페이지 섹션을 장식용 부유 카드로 만들지 않는다.
- 카드 안에 다시 카드를 중첩하지 않는다.
- 카드 모서리는 기본 `6px`, 페이지 섹션은 최대 `8px`를 사용한다.

### 8.6 표

- 헤더는 짧고 명사형으로 작성한다.
- 식별 열은 이름과 키를 함께 표시한다.
- 상태, 오너, 최근 수정일을 가능한 범위에서 제공한다.
- 행 작업은 `빠른 액션` 열에 모은다.
- 모바일에서 표 자체를 압축하지 않고 `.platform-table` 내부 스크롤을 사용한다.
- 빈 표 헤더만 남기지 말고 빈 상태와 다음 작업을 표시한다.

### 8.7 검증과 피드백

- 입력 오류: 해당 필드 아래
- 여러 조건의 사전 점검: 검증 패널
- 비동기 실행: 실행 중, 완료, 실패 상태를 같은 위치에서 갱신
- 시스템 오류: 사용자가 할 수 있는 다음 조치를 포함
- 성공 피드백: 무엇이 저장 또는 실행되었는지 명시

### 8.8 모달

- 사용 목적 확인, 비교, 편집 후 저장처럼 현재 맥락을 유지해야 하는 작업에 사용한다.
- 모달 제목은 대상과 작업을 함께 표현한다.
- 원본 개체 요약을 먼저 보여주고 입력을 받는다.
- 닫기, 취소, `Esc`를 지원한다.
- 긴 결과는 모달 내부에서 스크롤한다.

## 9. 화면별 필수 패턴

### 9.1 상품 팩토리

- 고객 맥락, 데이터 자산, 검토 및 생성의 3단계 위저드
- 고객군과 데이터 자산은 등록소 선택
- 민감 자산 선택 시 사용 목적 필수
- AI 후보는 비교, 채택, 수정 후 저장, 폐기를 제공

### 9.2 플로우 스튜디오

- 데이터, 변환과 AI, 거버넌스 노드 그룹
- 입력과 출력 노드 존재 여부 검증
- 도구와 에이전트 노드는 등록소 참조 필수
- 변경 비교 후 새 버전으로 저장
- 하단에 샘플 실행과 최근 실행 로그 제공

### 9.3 에이전트 빌더

- 입력, 프롬프트, LLM, RAG, 도구, 메모리, 승인, 가드레일, 출력 노드 표현
- 도구는 URL 직접 입력이 아니라 도구 등록소에서 선택
- 도구 위험도와 승인 필요 여부를 선택 즉시 표시
- 테스트 콘솔 결과와 실행 트레이스를 같은 화면에서 확인
- 고위험 실행은 승인 또는 정책 차단 상태를 명시

### 9.4 정책 시뮬레이션

- 점검 대상은 등록된 상품, 에이전트, 플로우, 도구, 데이터 자산에서 선택
- 대상 미리보기 후 민감도와 사용 조건 입력
- 결과는 판정, 위험 점수, 발견 항목, 필요 승인, 개선 조치로 구성

### 9.5 마켓플레이스

- 승인된 개체만 노출
- 카드에는 유형, 이름, 키, 버전, 설명, 오너, 구독 수 표시
- 접근 요청 모달에서 대상 요약, 사용 목적, 정책 범위 확인을 제공

## 10. 시각 토큰

토큰의 실제 구현 기준은 `internal/proxy/admin_ui.go`의 CSS 변수다.

| 토큰 | Light | 용도 |
| --- | --- | --- |
| `--bg` | `#f6f7f9` | 앱 배경 |
| `--panel` | `#ffffff` | 주요 표면 |

### 10.1 타이포그래피

- 기본 글꼴: 시스템 sans-serif와 한국어 시스템 글꼴
- 기본 크기: `14px`
- 섹션 제목: `13px`, 높은 굵기
- 카드 제목: `13px`에서 `15px`
- KPI 값: `22px`
- 자간은 음수로 설정하지 않는다.
- 화면 폭에 비례해 글자 크기를 확대하지 않는다.

### 10.2 간격과 크기

- 기본 컨트롤 높이: `38px`
- 기본 textarea 최소 높이: `96px`
- 라벨과 내부 요소 간격: `6px`
- 폼 행 간격: `18px`
- 기본 간격: `8px`, `10px`, `12px`, `14px`, `16px`
- 본문 최대 폭: `1440px`
- 고정 형식 요소는 grid track, min/max, aspect-ratio로 크기를 안정화한다.

## 11. 반응형 기준

지원 기준 화면:

- 데스크톱: `1440 x 900` 이상
- 소형 데스크톱·태블릿: `961px` 이상
- 모바일: `390 x 844`

`960px` 이하에서는 다음 규칙을 적용한다.

- KPI와 다열 그리드는 한 열로 전환
- 스튜디오는 팔레트, 캔버스, 속성 순서로 적층
- Product Twin 점수와 캔버스는 한 열로 전환
- 다음 작업 패널은 세로 배치
- 표는 컨테이너 내부 스크롤
- 문서 전체의 가로 스크롤은 허용하지 않음
- 버튼 텍스트와 긴 식별자는 줄바꿈 또는 `overflow-wrap` 적용

## 12. 접근성

- 모든 입력은 시각적 라벨 또는 접근 가능한 이름을 가진다.
- 키보드만으로 탐색, 선택, 저장, 모달 닫기가 가능해야 한다.
- 포커스 표시를 제거하지 않는다.
- 색상 외에 텍스트, 아이콘, 테두리로 상태를 구분한다.
- 오류는 필드와 연결되고 수정 방법을 설명한다.
- 표 헤더와 데이터 셀의 관계를 유지한다.
- 터치 대상은 모바일에서 겹치지 않아야 한다.
- 모달이 열리면 배경 작업과 시각적으로 분리한다.

## 13. 문구 기준

### 13.1 버튼

명확한 동사와 목적어를 사용한다.

- 좋음: `도구 등록`, `정책 사전 점검`, `새 버전 저장`, `접근 요청`
- 피함: `확인`, `처리`, `실행하기`, `계속`

### 13.2 오류

`무엇이 실패했는지 + 사용자가 할 수 있는 조치`

예시:

> 새 버전을 저장하지 못했습니다. 필수 참조와 노드 연결을 확인해 주세요.

### 13.3 빈 상태

`현재 없는 것 + 다음 작업`

예시:

> 등록된 플로우가 없습니다. 새 플로우를 만들어 첫 실행을 검증하세요.

### 13.4 확인 대화상자

대상, 영향, 감사 기록 여부를 설명한다. 단순히 “진행하시겠습니까?”만 사용하지 않는다.

## 14. 금지 패턴

- 기존 개체의 키나 이름을 자유 텍스트로 입력
- 선택 후 요약 없이 식별자만 표시
- AI 결과 자동 저장
- 상태를 색상만으로 표현
- 사용자에게 백엔드 영문 오류를 그대로 노출
- 카드 안에 카드 중첩
- 장식용 그라데이션, 구형 장식, 흐릿한 배경 이미지
- 운영 화면의 대형 히어로와 홍보성 레이아웃
- 모바일에서 문서 전체 가로 스크롤
- 저장, 삭제, 출시 결과를 `alert` 하나로만 전달
- 필수 입력을 placeholder로만 설명

## 15. 디자인 변경 절차

1. 변경하려는 사용자 문제와 대상 역할을 기록한다.
2. 기존 공통 패턴으로 해결 가능한지 확인한다.
3. 새 패턴이면 이 문서의 컴포넌트 계약을 먼저 갱신한다.
4. 정상, 빈 상태, 로딩, 오류, 차단, 권한 없음 상태를 설계한다.
5. 데스크톱과 모바일에서 구현한다.
6. Product Design과 해당 Domain Owner가 용어와 흐름을 검토한다.
7. 접근성, 브라우저 오류, 회귀 테스트를 통과한 뒤 배포한다.

- [ ] 브라우저 콘솔 오류와 경고가 없다.
